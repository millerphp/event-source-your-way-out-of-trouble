The audit trigger graveyard


CREATE TRIGGER tabs_audit AFTER UPDATE ON tabs
FOR EACH ROW INSERT INTO tabs_audit_log
  (tab_id, old_total, new_total, changed_at)
VALUES (OLD.id, OLD.total, NEW.total, NOW());
Most teams have tried to build audit trails this way. Show the trigger, then move to the next slide for why it rots.

Why it rots

  • Records that something changed. Not why.
  • Often disabled "temporarily" in 2021. Still disabled.
  • Diverges from the application logic over time.
They almost always rot. The triggers capture the wrong granularity — they record column changes, not business events. You end up reverse-engineering intent from diffs, which is exactly the problem we set out to solve.

The 4am hotfix


-- ticket #INC-4471, customer escalation, just do it
UPDATE tabs
SET total = 53.00, status = 'paid'
WHERE id = 7;

-- and the loyalty points
UPDATE customers SET points = points + 53 WHERE id = 412;

COMMIT;
-- TODO: tell finance on Monday
DANGER
Everyone has written this. Or worse: written something like this and forgotten the WHERE clause. The hotfix UPDATE is a confession that the system can't represent the correction as a business event. This is going to be the single biggest "before vs after" of the talk.

We patched the symptom

  • The total is now correct in the database
  • The reason the total was wrong is still in the codebase
  • Next Friday: same bug, different tab
  • The hotfix is now a forever undocumented exception to the data
The cost of CRUD hotfixes isn't just risk — it's that you're treating the data while the code keeps producing bad data. Event sourcing flips this: the fix lives in the event stream and the projection logic, and both are repeatable, replayable, and reviewable.

Event sourcing, defined

Store every state change as an immutable event.
The current state is a derivation, not the source of truth.

Events are facts. Facts don't change. Interpretations do.

Two important words: immutable, derivation. Immutable because the past doesn't change — Friday night's drink really was poured. Derivation because what you do with that fact (loyalty points, VAT calc, nightly takings) can change — and you can recompute it from the same fact tomorrow.

The mental flip. We don't UPDATE. We don't DELETE. We INSERT. Every change is a new row in the event log. State = fold(events)


$tab = new TabState();
foreach ($eventStore->streamFor('tab-7') as $event) {
    $tab = $tab->apply($event);
}

Every read is a replay. Storage is the log. Memory is the projection.

This is the engine of event sourcing in three lines. Everything else — projections, snapshots, time travel — is an optimisation or a variation on this loop. If they remember nothing else, remember this.

A command produces zero or more events.

The grammar matters. Commands are requests; they can fail validation ("tab is already paid"). Events are statements about what actually happened. Name your events in the past tense — it forces the right mental model.

Aggregate

  • The consistency boundary
  • One aggregate = one stream
  • Decides which commands are valid
  • Produces events as the only way to change state

In our pub: Tab is an aggregate. KitchenTicket is another.

The smallest thing you'd lock for one business operation. A tab — not the whole pub.

Aggregate is a Domain-Driven Design term. Don't get hung up on it. Practical heuristic: an aggregate is the smallest thing you'd want to lock for a single business operation. A tab. An order. A booking. Not "the whole pub."

Event store

  • Append-only log of events
  • Per-stream ordering guaranteed
  • Optimistic concurrency via stream version

It can be a database table. It really can.

We'll build one in PHP with PDO in Part 3.

De-mystify the event store. People think they need EventStoreDB or Kafka. They don't. A MySQL table with (stream_id, version, event_type, payload, recorded_at) and a unique constraint on (stream_id, version) is a perfectly adequate event store for most PHP shops.

Not a free lunch — the bottom two are real costs. The top six are why we pay them.

Be honest on the last two rows. We're not selling a free lunch. Storage cost is rarely actually a problem — events are tiny compared to logs or media. The learning curve is real and we'll address it in the strangling section.

What we get for free

  • Complete audit trail (with intent, not just diffs)
  • Time travel — state at any point in the past
  • Multiple read models from one source
  • Test fixtures = real production sequences
  • Postmortems that read like English
  • "Why did this happen?" becomes answerable
Hammer the benefits. Especially "test fixtures = real sequences" — being able to capture a production bug as the actual event stream that produced it, and replay it locally with a unit test, is genuinely a superpower. We'll demo that in Part 4.

What it costs

  • Eventual consistency between writes and read models
  • A genuinely new mental model for the team
  • More storage (cheap, but not zero)
  • Event schema evolution is a real discipline
  • Tooling is less mature than CRUD in PHP-land
Eventual consistency is the one that bites people. The write goes "I added a drink" and the projection takes 50ms to catch up. UIs need to handle this. We'll touch on patterns for it. Schema evolution we'll cover in detail in Part 5 — upcasters.

Three myths

  • ✗ Myth: You need Kafka / Axon / a special database. ✓ Truth: a MySQL table works fine.
  • ✗ Myth: You have to event-source the whole system. ✓ Truth: one bounded context is enough.
  • ✗ Myth: It's only worth it for banks and huge systems. ✓ Truth: one pub tab is plenty.
These come up in Q&A every time. Address them now and you save 20 minutes later. The "MySQL is fine" point is especially important for PHP shops who don't want a new ops dependency.

A base event interface


namespace CrookedAnchor\Events;

interface DomainEvent
{
    public function aggregateId(): string;
    public function occurredAt(): \DateTimeImmutable;
    public function eventType(): string;
    public function payload(): array;
}

Four methods. That's the whole contract.

Keep the interface dead simple. Some libraries add more (metadata, causation IDs, correlation IDs) — we'll mention those later. For now, four methods is plenty to make the rest work.

An event, in full


final class DrinkAddedToTab implements DomainEvent
{
    public function __construct(
        private readonly string $tabId,
        private readonly string $item,
        private readonly int $priceInPence,
        private readonly string $addedBy,
        private readonly \DateTimeImmutable $occurredAt,
    ) {}

    public function aggregateId(): string { return $this->tabId; }
    public function occurredAt(): \DateTimeImmutable { return $this->occurredAt; }
    public function eventType(): string { return 'drink_added_to_tab'; }

    public function payload(): array
    {
        return [
            'item' => $this->item,
            'price_in_pence' => $this->priceInPence,
            'added_by' => $this->addedBy,
        ];
    }

    public function item(): string { return $this->item; }
    public function priceInPence(): int { return $this->priceInPence; }
}
Immutable. Readonly properties. Plain PHP 8.1+. The payload() method is what we'll serialise to the event store. The named getters are what the aggregate uses to apply the event. Note: prices in pence — never store money as floats. Burned-in good habit.

The aggregate root


abstract class AggregateRoot
{
    /** @var DomainEvent[] */
    private array $pendingEvents = [];
    private int $version = 0;

    public function version(): int
    {
        return $this->version;
    }

    /** @return DomainEvent[] */
    public function releaseEvents(): array
    {
        $events = $this->pendingEvents;
        $this->pendingEvents = [];
        return $events;
    }
		

    protected function recordThat(
        DomainEvent $event
    ): void {
        $this->apply($event);
        $this->pendingEvents[] = $event;
        $this->version++;
    }

    public function replay(iterable $events): void
    {
        foreach ($events as $event) {
            $this->apply($event);
            $this->version++;
        }
    }

    abstract protected function apply(
        DomainEvent $event
    ): void;
}
The heart of the pattern. Two ways events arrive: recordThat for new events produced by a command, and replay for events loaded from the store. Both go through apply. The pendingEvents array is what gets persisted at the end of the request.

The Tab aggregate (1/5)


final class Tab extends AggregateRoot
{
    private string $tabId;
    private string $tableLabel;
    private string $status = 'open'; // open | split | paid | voided
    private int $runningTotalPence = 0;
    /** @var array<string, int> itemId => quantity */
    private array $unservedFood = [];
    private array $allergenWarnings = [];

    public static function open(
        string $tabId, string $tableLabel, string $server
    ): self {
        $tab = new self();
        $tab->recordThat(new TabOpened(
            $tabId, $tableLabel, $server, new \DateTimeImmutable()
        ));
        return $tab;
    }

    public static function fromHistory(iterable $events): self
    {
        $tab = new self();
        $tab->replay($events);
        return $tab;
    }
Two named constructors. open creates a brand new tab and emits the first event. fromHistory is what the repository uses to rehydrate a tab from the store. Notice we never expose a regular constructor — you can't accidentally create a Tab in an inconsistent state.

The Tab aggregate (2/5)


    public function addDrink(string $item, int $priceInPence, string $by): void
    {
        $this->ensureOpen();
        $this->recordThat(new DrinkAddedToTab(
            $this->tabId, $item, $priceInPence, $by, new \DateTimeImmutable()
        ));
    }

    public function orderFood(
        string $item, int $priceInPence, array $allergens, string $by
    ): void {
        $this->ensureOpen();
        $this->recordThat(new FoodOrderedToTab(
            $this->tabId, $item, $priceInPence, $allergens, $by,
            new \DateTimeImmutable()
        ));
        foreach ($allergens as $a) {
            $this->recordThat(new AllergenWarningRecorded(
                $this->tabId, $item, $a, new \DateTimeImmutable()
            ));
        }
    }
The command methods. Each validates business rules before recording the event — once an event is recorded, it's a fact, so validation has to be a guard up front. Note orderFood fans out an AllergenWarningRecorded event per allergen.

The Tab aggregate (3/5)


    public function pay(string $method, int $amountInPence): void
    {
        $this->ensureOpen();
        if ($amountInPence < $this->runningTotalPence) {
            throw new \DomainException('Underpayment for tab.');
        }
        $this->recordThat(new TabPaid(
            $this->tabId, $method, $amountInPence, new \DateTimeImmutable()
        ));
    }

    private function ensureOpen(): void
    {
        if ($this->status !== 'open') {
            throw new \DomainException("Tab is {$this->status}, cannot modify.");
        }
    }
pay guards against underpayment before recording TabPaid. ensureOpen is the invariant every command calls first — it's what protects against modifying a closed tab.

The Tab aggregate (4/5)


    protected function apply(DomainEvent $event): void
    {
        match (true) {
            $event instanceof TabOpened           => $this->whenOpened($event),
            $event instanceof DrinkAddedToTab     => $this->whenDrinkAdded($event),
            $event instanceof FoodOrderedToTab    => $this->whenFoodOrdered($event),
            $event instanceof FoodServed          => $this->whenFoodServed($event),
            $event instanceof AllergenWarningRecorded
                                                  => $this->whenAllergenWarning($event),
            $event instanceof TabPaid             => $this->whenPaid($event),
            $event instanceof TabVoided           => $this->whenVoided($event),
            default => null, // unknown / future events are ignored
        };
    }

    private function whenOpened(TabOpened $e): void
    {
        $this->tabId = $e->aggregateId();
        $this->tableLabel = $e->tableLabel();
    }
The dispatcher: apply() routes each event to its when* handler via match. The default => null matters — it lets you safely deploy code that meets new event types it doesn't yet know how to handle. whenOpened sets the tab's identity from the first event.

    {

    }

The state-change methods. Pure functions of (state, event) → new state. No side effects, no IO, no dependencies — that's what makes them safe to call on either new events or replayed events.

interface EventStore
{
    /**
     * @param DomainEvent[] $events
     * @throws ConcurrencyException
     */
    public function append(
        string $streamId,
        int $expectedVersion,
        array $events,
    ): void;

    /** @return iterable<DomainEvent> */
    public function loadStream(string $streamId): iterable;

    /** @return iterable<DomainEvent> */
    public function loadAllFrom(int $globalPosition = 0): iterable;
}

expectedVersion = optimistic concurrency control.

Three operations. Append (with expected version to detect concurrent writes). Load one stream (for rehydrating an aggregate). Load all events from a global position (for building projections). That's the whole contract. Everything else is implementation detail.

The event store table


CREATE TABLE events (
    global_position BIGINT       NOT NULL AUTO_INCREMENT,
    stream_id       VARCHAR(64)  NOT NULL,
    stream_version  INT          NOT NULL,
    event_type      VARCHAR(64)  NOT NULL,
    payload         JSON         NOT NULL,
    occurred_at     DATETIME(6)  NOT NULL,
    recorded_at     DATETIME(6)  NOT NULL DEFAULT CURRENT_TIMESTAMP(6),

    PRIMARY KEY (global_position),
    UNIQUE KEY uq_stream_version (stream_id, stream_version),
    INDEX idx_event_type (event_type),
    INDEX idx_stream (stream_id)
) ENGINE=InnoDB;

One table. That's your event store.

The unique constraint on (stream_id, stream_version) is the optimistic concurrency mechanism. Two writers trying to append version 5 to the same stream — one of them gets a duplicate key error. That's how we detect a conflict. global_position gives projections a totally-ordered global stream to consume.

PDO event store (1/2)


final class PdoEventStore implements EventStore
{
    public function __construct(
        private readonly \PDO $pdo,
        private readonly EventSerializer $serializer,
    ) {}

    public function append(string $streamId, int $expectedVersion, array $events): void
    {
        $this->pdo->beginTransaction();
        try {
            $stmt = $this->pdo->prepare(
                'INSERT INTO events
                   (stream_id, stream_version, event_type, payload, occurred_at)
                 VALUES (:sid, :ver, :type, :payload, :occurred)'
            );
            $version = $expectedVersion;
            foreach ($events as $event) {
                $version++;
                $stmt->execute([
                    ':sid'      => $streamId,
                    ':ver'      => $version,
                    ':type'     => $event->eventType(),
                    ':payload'  => json_encode($event->payload(), JSON_THROW_ON_ERROR),
                    ':occurred' => $event->occurredAt()->format('Y-m-d H:i:s.u'),
                ]);
            }
            $this->pdo->commit();
        } catch (\PDOException $e) {
            $this->pdo->rollBack();
            if ($e->getCode() === '23000') { // duplicate key
                throw new ConcurrencyException(
                    "Stream {$streamId} expected v{$expectedVersion}, conflict."
                );
            }
            throw $e;
        }
    }
Append in a transaction. Increment the version per event. Duplicate key on (stream_id, stream_version) is the concurrency signal — translate it into a domain-meaningful exception. The caller (the repository) decides whether to retry.

PDO event store (2/2)


    public function loadStream(string $streamId): iterable
    {
        $stmt = $this->pdo->prepare(
            'SELECT event_type, payload, occurred_at
             FROM events
             WHERE stream_id = :sid
             ORDER BY stream_version ASC'
        );
        $stmt->execute([':sid' => $streamId]);
        while ($row = $stmt->fetch(\PDO::FETCH_ASSOC)) {
            yield $this->serializer->deserialize(
                $row['event_type'],
                json_decode($row['payload'], true, 512, JSON_THROW_ON_ERROR),
                new \DateTimeImmutable($row['occurred_at']),
                $streamId,
            );
        }
    }

    public function loadAllFrom(int $globalPosition = 0): iterable
    {
        $stmt = $this->pdo->prepare(
            'SELECT global_position, stream_id, event_type, payload, occurred_at
             FROM events
             WHERE global_position > :pos
             ORDER BY global_position ASC'
        );
        $stmt->execute([':pos' => $globalPosition]);
        while ($row = $stmt->fetch(\PDO::FETCH_ASSOC)) {
            yield $row['global_position'] => $this->serializer->deserialize(
                $row['event_type'],
                json_decode($row['payload'], true, 512, JSON_THROW_ON_ERROR),
                new \DateTimeImmutable($row['occurred_at']),
                $row['stream_id'],
            );
        }
    }
}
Generators (yield) so we don't load the whole stream into memory. The serializer is responsible for mapping (event_type, payload) back to a typed DomainEvent. We won't show the serializer — it's a switch statement, you can imagine it.

The repository


final class TabRepository
{
    public function __construct(private readonly EventStore $store) {}

    public function load(string $tabId): Tab
    {
        $events = iterator_to_array($this->store->loadStream("tab-{$tabId}"));
        if ($events === []) {
            throw new \RuntimeException("Tab {$tabId} not found.");
        }
        return Tab::fromHistory($events);
    }

    public function save(Tab $tab): void
    {
        $newEvents = $tab->releaseEvents();
        if ($newEvents === []) {
            return;
        }
        $expectedVersion = $tab->version() - count($newEvents);
        $this->store->append(
            "tab-{$tab->id()}",
            $expectedVersion,
            $newEvents,
        );
    }
}
The repository is the bridge between aggregate and event store. Notice how thin it is. load streams the events, rehydrates the aggregate. save takes the pending events and appends them. Expected version is "what version was I at before I started?" — so the store can detect concurrent edits.

The cashier's projection


final class CashierProjection
{
    public function __construct(private readonly \PDO $readDb) {}

    public function handle(DomainEvent $event): void
    {
        match (true) {
            $event instanceof TabOpened          => $this->onOpened($event),
            $event instanceof DrinkAddedToTab    => $this->addToTotal($event->aggregateId(), $event->priceInPence()),
            $event instanceof FoodOrderedToTab   => $this->addToTotal($event->aggregateId(), $event->priceInPence()),
            $event instanceof DiscountApplied    => $this->addToTotal($event->aggregateId(), -$event->amountInPence()),
            $event instanceof TabPaid            => $this->markPaid($event),
            default                              => null,
        };
    }

    private function onOpened(TabOpened $e): void
    {
        $this->readDb->prepare(
          'INSERT INTO tab_summary
             (tab_id, table_label, status, total_pence, opened_at)
           VALUES (?, ?, "open", 0, ?)'
        )->execute([$e->aggregateId(), $e->tableLabel(), $e->occurredAt()->format(DATE_ATOM)]);
    }

    private function addToTotal(string $tabId, int $delta): void
    {
        $this->readDb->prepare(
          'UPDATE tab_summary SET total_pence = total_pence + ? WHERE tab_id = ?'
        )->execute([$delta, $tabId]);
    }
}
The projection's job: turn events into row updates on a read table. It owns its own database (or schema, or table set). Note: this can use UPDATE — the projection table isn't the source of truth, it's a cache we can rebuild any time.

How projections run


// A simple catch-up subscriber: poll, dispatch, checkpoint
final class Catchup
{
    public function __construct(
        private readonly EventStore $store,
        private readonly array $projections, // [name => handler]
        private readonly Checkpoints $checkpoints,
    ) {}

    public function tick(): void
    {
        foreach ($this->projections as $name => $projection) {
            $position = $this->checkpoints->get($name);
            foreach ($this->store->loadAllFrom($position) as $newPos => $event) {
                $projection->handle($event);
                $this->checkpoints->set($name, $newPos);
            }
        }
    }
}

Run via cron, supervisor, or a long-lived worker.

The catch-up subscriber pattern. Each projection has its own checkpoint — its position in the global stream. On each tick, fetch new events from where it left off, process them, save the checkpoint. Idempotent. Crash-safe. Boring. Good.

Part 5

The safety net

Pivot to the second big payoff: how do you actually fix the bug in production once you've found it? This is where the discipline of "events are immutable" forces good habits.

The wrong way


UPDATE tabs SET total = 5300 WHERE id = 7;
-- the symptom is fixed
-- the code that produced it is unchanged
-- the audit trail says nothing
-- the customer escalation is closed
-- the bug ships again next Friday
DON'T
The familiar villain. The hotfix UPDATE is fast, satisfying, and corrosive. It hides the bug, removes the evidence, and makes the next investigation harder. We're going to do better.

The right way: compensating event


$tab = $tabs->load('tab-2026-05-15-7');

$tab->adjustTotal(
    reason: 'happy-hour discount under-applied (INC-4471)',
    deltaPence: -670,
    by: 'christopher@phparch.com',
);

$tabs->save($tab);

// Emits: TabAdjusted {
//   tab: 'tab-2026-05-15-7',
//   delta: -670,
//   reason: 'happy-hour discount under-applied (INC-4471)',
//   by: 'christopher@phparch.com',
//   occurred_at: '2026-05-16T09:12:01Z'
// }

The fix is itself an event. It's auditable. It's reversible. It's history.

Compensating events. You don't change the past — Friday night really did charge £6.70 too much. You add a corrective event on Saturday morning. The customer's refund is now a first-class business event. Finance can see it. The auditor can see it. The customer can see it on their statement.

Why we don't mutate history

  • Replay determinism — the same stream must always produce the same state
  • Other projections may have already consumed the wrong event
  • Audit integrity — you can't quietly rewrite financial history
  • Trust — if events can change, they aren't facts anymore

Edit an event and you've reinvented CRUD.

This is the rule that takes the longest to internalise. Engineers want to "just fix the bad event." Don't. The whole value of the model is that events are facts. Once you allow editing, you've reinvented CRUD.

The fix loop

  1. 1Identify the bad event(s) in the stream
  2. 2Write the corrected logic in the aggregate / projection
  3. 3Add a compensating event (or rebuild a projection)
  4. 4Replay the affected streams to verify
  5. 5Deploy

The future

prevented by the new logic

The past

corrected by compensating events

Neither touches old data. Both ship in a normal PR.

The biggest idea in this section — why we do all of this. Two complementary moves: prevent future bugs by fixing the code, and correct historical data with new events. Land the two-column payoff hard: the future is protected AND the past is put right, without ever editing old data. Both reviewable in a normal PR.

Rebuilding a projection


// Logic in CashierProjection was wrong all along
// Don't UPDATE the tab_summary table — rebuild it

$readDb->exec('TRUNCATE TABLE tab_summary');
$checkpoints->reset('cashier');

// then on the next catch-up tick:
$catchup->tick();
// => replays every event ever recorded
// => tab_summary is now correct

For projections, you can rewrite history because it isn't history. It's a cache.

Critical distinction. Events = immutable history. Projections = derived caches. You can rebuild a projection any time, any way. That's what makes deploying a new pricing engine safe: shadow project it from real events and inspect the result before flipping the read.

But the event shape changed!

The case where you really do need to evolve an event:


// v1 of the event (2024)
DrinkAddedToTab { tab, item, price }

// v2 of the event (2026): we now need VAT rate per item
DrinkAddedToTab { tab, item, price, vat_rate }

Two options. Add a new event type, or upcast on read.

Schema evolution is the most legitimate "but how do I…" question. Two answers. Add a new event type (DrinkAddedToTabV2) — simple, additive, no backfill. Or upcast on read — transform old events into the new shape as they're loaded. We'll show upcasting.

Upcasting


final class DrinkAddedUpcaster
{
    public function upcast(array $payload, int $payloadVersion): array
    {
        if ($payloadVersion < 2) {
            // v1 didn't have vat_rate — infer it from the item
            $payload['vat_rate'] = $this->inferVatFor($payload['item']);
            $payloadVersion = 2;
        }
        return $payload;
    }
}

Runs in the serializer on every load. Old events look new to new code.

Upcasters live in the deserialization path. They never touch stored data. The historical event in the event store is still v1 — we just transform it into v2 every time we read it. Storage stays immutable; code sees the latest shape.

When to upcast vs add a new event

Add a new event typeUpcast
Meaning changedShape changed, meaning didn't
New field is fundamentally new infoNew field is derivable from old
You want old events to still be readable as-isYou want one code path

Meaning changed? New event.  Just the shape? Upcast.

Rule of thumb. If the business meaning of the event is genuinely different, give it a new name — DrinkAddedToTabWithVat. If it's the same event with a richer payload, upcast. Don't upcast to paper over semantic changes; that's how you get subtle bugs.

Deploy with confidence

  1. 1Write the fix locally
  2. 2Replay last week's production stream through the new code
  3. 3Diff projection outputs against current production
  4. 4If the diff is exactly the bugs you intended to fix — ship

You're not guessing. You're testing against real history.

This is the deployment story I love most. You can replay a week of production traffic against your fix, offline, before it ever sees a live request. The "I'm pretty sure this is right" deploy gets replaced by "I've already seen it produce the right answer for 47,892 real tabs."

  • Where are the manual UPDATEs?
  • Where do auditors / finance complain?
  • Where is the code most afraid?

For the Crooked Anchor: tabs and payments.

Don't start with the easy module. Start with the one that hurts. ES pays back proportionally to how much "where did the money go" investigation you're currently doing. The boring CRUD admin screens? Leave them.

Step 2: identify the events

Use Event Storming:

  • Write past-tense business facts
  • Arrange them on a timeline

— Alberto Brandolini, EventStorming workshop format

Event Storming is the discovery technique that pairs with ES. It's deliberately low-tech — sticky notes, a long wall, no laptops. The output is a shared vocabulary. If you can't agree what your business events are, you can't event-source it. This step is often where teams find they've been arguing about two different domain models all along.

Step 3: write twice

1 Pain 2 Events 3 Write twice 4 Backfill 5 Read flip 6 Write flip 

public function payTab(int $tabId, string $method, int $amount): void
{
    // Legacy path — still authoritative
    $this->legacyTabService->markPaid($tabId, $method, $amount);

    // Shadow event log — building the future
    try {
        $tab = $this->tabRepository->loadOrInitialize($tabId);
        $tab->pay($method, $amount);
        $this->tabRepository->save($tab);
    } catch (\Throwable $e) {
        // log but don't fail the request — we're still in shadow mode
        $this->logger->warning('Shadow ES write failed', ['e' => $e]);
    }
}
The dual-write phase. Legacy is still the source of truth — if the shadow write fails, the customer doesn't notice. You're building up an event log alongside the existing data. Compare them daily. When they agree for long enough that you trust the shadow, flip the source of truth.

Step 4: backfill from history

// Generate events from existing CRUD rows as best you can
foreach ($legacyDb->query('SELECT * FROM tabs ORDER BY opened_at') as $row) {
    $tab = Tab::open($row['id'], $row['table_label'], $row['server']);
    foreach ($legacyDb->query("SELECT * FROM tab_items WHERE tab_id = ?", [$row['id']]) as $item) {
        // best-effort reconstruction — missing fields get sentinel values
        $tab->addDrink($item['name'], $item['price_pence'], 'backfill');
    }
    if ($row['status'] === 'paid') {
        $tab->pay($row['payment_method'] ?? 'unknown', $row['total_pence']);
    }
    $tabRepository->save($tab);
}

Mark these events with source: 'backfill'. Be honest about uncertainty.

Backfilling is imperfect — by definition, you don't have the data the CRUD model discarded. That's fine. Mark backfilled events as such. New events from today onward are precise; historical ones are best-effort. It's still better than not having a stream at all.

Step 5: flip the reads

public function getTabSummary(int $tabId): array
{
    if ($this->featureFlags->enabled('es_cashier_read', $tabId)) {
        return $this->cashierProjection->findByTabId($tabId);
    }
    return $this->legacyTabService->getSummary($tabId);
}

Roll out by percentage. Compare. Fix discrepancies. Increase the percentage.

Feature-flag the read switch. Start with 1% of tabs reading from the projection, compare against legacy, grow the percentage as confidence builds. The dual-write was building the data; the read flip is when users start seeing ES-derived numbers.

Step 6: flip the writes

1 Pain 2 Events 3 Write twice 4 Backfill 5 Read flip 6 Write flip
  • The shadow event log has been authoritative-shaped for weeks
  • Legacy writes still happen, but only as a fallback / verification
  • Remove the legacy write call. The event log is the source of truth.
  • Eventually: drop the legacy table.
The final step. By the time you remove the legacy write, you've been operating with two copies for weeks or months and they've agreed. The legacy table becomes vestigial; eventually you drop it. The bounded context is now fully event-sourced. Onto the next one.

Snapshots, for when streams grow


// A long-lived tab with thousands of events?
// Snapshot the state periodically, replay only the tail.

$snapshot = $snapshots->latestFor($streamId);
if ($snapshot) {
    $tab = Tab::fromSnapshot($snapshot);
    $tail = $store->loadStreamFrom($streamId, $snapshot->version());
    $tab->replay($tail);
} else {
    $tab = Tab::fromHistory($store->loadStream($streamId));
}

Premature snapshotting is the root of much complexity. Wait until you need it.

The "but won't replay be slow" question. For most aggregates, no — a tab has tens of events, not millions. For genuinely long-lived aggregates (a year-long subscription, a multi-decade account), snapshot when measurement says it matters. Not before.

When NOT to event source

  • Read-mostly catalogues (products, content)
  • Pure lookup tables (postcodes, currencies)
  • Reporting warehouses (already denormalised)
  • Anywhere the "why" of changes doesn't matter
  • Anywhere you don't control the model (3rd-party SaaS)

If nobody would ever ask "how did we get here?" — CRUD is fine.

Honest scope-setting. Most apps have a mix. The tabs/payments/orders bounded contexts benefit; the product catalogue doesn't. Don't be religious. Use ES where the "why" matters.

Team adoption tips

  • Vocabulary first — Event Storm before you code
  • One context at a time — resist big-bang
  • Pair on the first aggregate
  • Code review ritual: "past tense? a real business fact?"
  • Postmortem template: list the events
  • Celebrate the first bug that's a one-line projection fix
